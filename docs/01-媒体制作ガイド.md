# 媒体制作ガイド(制作標準)

全教材はこのガイドに従って制作する。品質と統一感を担保するための「制作の憲法」。

---

## 1. 解説動画(M1)制作標準

### 1.1 基本仕様

| 項目 | 仕様 |
|---|---|
| 長さ | 5〜15分(15分を超える場合は分割) |
| 解像度 | 1920×1080 / 16:9(ドリル用ショート動画のみ 9:16 可) |
| 構成 | OP(15秒)→ 今日のゴール提示 → 本編(3〜5トピック)→ まとめ(要点3つ)→ 次回予告 |
| テロップ | ナレーション要約を常時下部表示(ミュート視聴でも理解できること) |
| 図解 | 1トピックにつき最低1枚のアニメーション図解 |

### 1.2 ナレーター「いろは」音声仕様

**キャラクター設定**: 新人の隣の席にいる、教えるのが大好きな先輩(20代女性)。
明るく、かわいらしく、しかし技術的には正確。

| 項目 | 指定 |
|---|---|
| 声質 | **かわいい系の女性ボイス**。やや高め・クリア・柔らかい響き |
| 話速 | ゆっくりめ(300〜330文字/分)。専門用語の前後はさらに遅く |
| トーン | 明るい。文末を沈めず、軽く上げる。疲れた声・平板読みは禁止 |
| 合成音声を使う場合 | 若年女性・明るい・囁かない、のプリセットを選択し、話速 -10%、ピッチ +5% を基準に調整 |

**口調ルール**:
- 一人称「わたし」、受講者の呼びかけは「みなさん」または「あなた」
- 「〜ですよ」「〜してみましょう!」「できましたか?」と語りかける
- 決め台詞: 章の重要ポイントで「**ここ、テストに出ますよ!**」/ 締めは「**今日もおつかれさまでした!**」
- 専門用語の直後に必ず一言フォロー:
  - 例「DHCP、つまり“IPアドレスの自動配布係”ですね」
  - 例「リバースプロキシ——“受付窓口サーバー”のことです」
- 禁止: 上から目線(「当然〜」「簡単ですよね」)、過度な擬音、意味のない笑い

### 1.3 台本テンプレート

台本は `videos/scripts/` に Markdown で置く。1シーン=1行のテーブル形式。

```markdown
# 【動画台本】<パート番号-章番号> <タイトル>(目標尺: X分)

## メタ情報
- 対象章 / 前提知識 / 学習ゴール(3つまで)

## 絵コンテ
| # | 尺 | 画面(ビジュアル指示) | ナレーション(全文) | テロップ |
|---|----|----|----|----|
```

- ナレーションは**一字一句の完成原稿**で書く(収録時に読み上げるだけの状態)。
- 画面指示は「何が・どう動くか」まで書く(例: 「PCアイコンからルーターへ矢印が伸びるアニメ」)。

## 2. HTML教材(M2)制作標準

| 項目 | 仕様 |
|---|---|
| 1ページの分量 | 読了10〜20分。超える場合はページ分割 |
| 構成 | ①この章で学ぶこと(箇条書き3〜5)→ ②本文 → ③まとめ → ④クイズへのリンク |
| 見出し | h2 は最大6個。h2 ごとに図または表を最低1つ |
| コード/設定例 | 必ずコピー可能なコードブロック。**1行ずつの日本語注釈**を付ける |
| 文体 | です・ます調。1文60文字以内目安。カタカナ語には初出時に説明 |
| デザイン | 共通CSS(ライト/ダーク自動対応)。スマホ閲覧対応(テーブルは横スクロール) |

**「読ませない」工夫(必須)**:
- 5画面スクロールごとに図解・表・演習のいずれかを挟む
- 重要語は `<strong>` + マーカー装飾で視線を誘導
- 「やってみよう」ボックス(手を動かす小演習)を1ページに1つ以上

## 3. クイズ(M3)制作標準

| 項目 | 仕様 |
|---|---|
| 形式 | ①4択 ②複数選択 ③穴埋め(テキスト入力) ④並べ替え |
| 1セット | 5〜20問。1問30秒〜1分で解ける粒度 |
| 解説 | **全問に解説必須**。正解の理由+間違い選択肢がなぜ違うかまで書く |
| 難易度 | ★基礎(用語)/ ★★標準(理解)/ ★★★実践(状況判断)を混ぜる。比率 3:5:2 |
| 合格基準 | 章末クイズ 70%、修了テスト 80% |

**良問の条件**:
- 「暗記」でなく「現場の判断」を問う(例: ×「pingのオプションは?」→ ○「社内サイトだけ見られない。最初に打つコマンドは?」)
- 選択肢は全て「ありそうな」もの(明らかなハズレ選択肢を作らない)
- 引っかけのための引っかけは禁止

クイズデータは `quizzes/` に JS ファイルで置き、共通エンジン(`portal/quiz.html`)で再生する。

```js
// クイズデータ形式
window.QUIZ_DATA = {
  id: "part3-ch05",
  title: "viエディタ 章末クイズ",
  passLine: 70,
  questions: [
    {
      type: "single",            // single | multi | fill | order
      level: 2,                   // 1〜3(★の数)
      q: "問題文",
      choices: ["A", "B", "C", "D"],
      answer: [1],                // choices の index(fill は文字列配列)
      explain: "解説(正解の理由と誤答の理由)"
    }
  ]
};
```

## 4. ハンズオンラボ(M4)制作標準

| 項目 | 仕様 |
|---|---|
| 環境 | 受講者のPC1台で完結(VirtualBox / WSL2 / 各クラウド無料枠)。会社の本番環境は絶対に使わせない |
| 構成 | ①ゴールと完成図 → ②所要時間・前提 → ③手順(番号付き)→ ④確認チェックリスト → ⑤うまくいかない時(FAQ)→ ⑥後片付け |
| 手順 | 1ステップ=1操作。実行コマンドと**期待される出力例**を必ず併記 |
| つまずき対策 | 各ステップに「よくあるエラーと対処」を折りたたみで用意 |
| 後片付け | 環境を消す/戻す手順まで書く(次のラボに影響させない) |

## 5. ドリル(M5)・ケーススタディ(M6)制作標準

### ドリル
- 1回5分・10問前後・スコアと過去最高記録を表示(localStorage)
- 対象: viキー操作 / サブネット計算 / コマンド当て / ポート番号 / ログ読解
- 出題はランダム順。同じ問題でも繰り返す価値がある反射系に限定

### ケーススタディ(分岐シナリオ)
- 舞台は統一世界観「株式会社ミライ商事」(従業員300名、拠点3つ、あなたは情シス2人目)
- 形式: 状況提示 → 選択肢3〜4 → 選択による結果分岐 → 振り返り解説
- **正解が1つとは限らない設計**にする(ベスト/次善/悪手 の3段階評価)
- 悪手を選んだ場合も「なぜ悪手か」を体験させてから解説する(即ゲームオーバーにしない)

## 6. 用語・表記統一

| ルール | 例 |
|---|---|
| カタカナ語末尾の長音は付ける | サーバー、ルーター、ユーザー |
| コマンド・ファイル名は等幅表記 | `systemctl`、`/etc/nginx/nginx.conf` |
| 略語は初出でフルスペル+日本語 | 「DNS(Domain Name System:名前解決の仕組み)」 |
| バージョン表記 | 教材内の手順は Ubuntu 24.04 LTS / Windows Server 2022 / nginx 1.24 系に統一 |
| 架空の値 | 例示ドメインは `example.co.jp`、社内IPは `192.168.10.0/24`、人名は「未来さん・匠さん」 |
