# Part 7 IT運用とヘルプデスク(全7章 / 約20時間 / レベル4)

## パート概要

技術を「継続的なサービス」として提供する力を養うパート。
ITIL の考え方を土台に、ヘルプデスク・資産管理・監視・バックアップという
情シスの定常業務を設計・改善できるレベルに引き上げる。

**到達目標**: 問い合わせ対応から障害対応までのプロセスを設計・運用でき、
監視とバックアップを「復旧できること」まで含めて構築できる。

## 各章詳細

### 7-1 ITIL入門(サービスとしてのIT)

- **学習ゴール**: インシデント管理・問題管理・変更管理の違いを実例で説明でき、SLAの概念を理解する
- **教材構成**
  - 動画「なぜ“いい人対応”では回らないのか」(12分): 属人対応が破綻する様子をドラマで→プロセス化の価値
  - クイズ: 10問(「これはインシデント?問題?変更?」の分類問題)
- **内容トピック**: インシデント/問題/変更/リリース管理 / SLA・OLA / ナレッジ管理 / チケット駆動運用(チケットに書いていないことは存在しない)

### 7-2 ヘルプデスク実践

- **学習ゴール**: 問い合わせの受付〜クローズまでを、優先度判断とエスカレーションを含めて回せる
- **教材構成**
  - ケーススタディ「ヘルプデスク繁忙日シミュレーション」(分岐型): 午前中に7件の問い合わせが同時に来る。優先度付け→切り分け→対応/エスカレーションを時間管理付きで体験(役員のPC不調 vs 全社系障害 vs 新人の操作質問…)
  - HTML「ヒアリングの技術」: 非ITユーザーの曖昧な訴え(「ネットが変」)を事実に翻訳する質問集
  - HTML「回答テンプレート集」: 感じの良い定型文(受付・進捗・クローズ・できない旨の伝え方)
- **内容トピック**: 優先度マトリクス(影響度×緊急度)/ 一次切り分けの標準 / エスカレーション基準 / FAQ・ナレッジベースの育て方 / 「自己解決率」というKPI

### 7-3 IT資産管理とライセンス管理

- **学習ゴール**: ハード・ソフト・ライセンスの台帳を設計し、棚卸しプロセスを回せる
- **教材構成**
  - HTML「資産管理入門」: 台帳項目設計、ラベル運用、資産管理ツールの位置づけ
  - 演習「台帳設計ワーク」: ミライ商事の資産リスト(意図的に不備あり)を渡し、問題点の指摘と台帳の再設計をさせる
  - クイズ: 8問(ライセンス違反ケースの判定問題)
- **内容トピック**: 現物管理とライフサイクル / ソフトウェアライセンスの型(デバイス/ユーザー/サブスク)/ 監査リスク / リース・レンタル・購入の使い分け / 廃棄時のデータ消去(証明書)

### 7-4 監視とアラート

- **学習ゴール**: 「何を・なぜ監視するか」を設計でき、監視ツールで死活・リソース監視を構築できる
- **教材構成**
  - ラボ「はじめての監視構築」(120分): Part 3 のVMに Zabbix(または Uptime Kuma で軽量代替)を導入 → 死活監視 → CPU/ディスク閾値アラート → 意図的に負荷をかけて発報体験 → 通知先設定
  - HTML「監視設計の考え方」: 監視項目の選び方、閾値設計、オオカミ少年アラート問題
- **内容トピック**: 死活・リソース・ログ・外形監視 / 閾値とアラート疲れ / エスカレーションフロー / 「監視されていないシステムは止まっていても気づけない」

### 7-5 バックアップとBCP/DR

- **学習ゴール**: 3-2-1ルールでバックアップを設計し、**リストア訓練** の重要性を体得。RPO/RTOで経営と会話できる
- **教材構成**
  - HTML「バックアップ設計教科書」: 3-2-1ルール、世代管理、ランサム対策(イミュータブル/オフライン)
  - 演習「リストア訓練」: Part 3 で作ったバックアップから実際に復元し、復元手順書を書く(「バックアップはあるのに戻せない」を体験的に予防)
  - 演習「BCP机上訓練」: 「本社ビルが計画停電。業務をどう継続する?」のワークシート
- **内容トピック**: RPO/RTO / フル・差分・増分 / バックアップの監視(成功していると思い込まない)/ DRサイトとクラウド活用 / 安否確認と連絡網

### 7-6 障害対応総合訓練

- **学習ゴール**: 障害発生から復旧報告までの一連の指揮を、時間圧の中で実行できる
- **教材構成**
  - **インタラクティブ演習「ミライ商事 障害対策室」(分岐型・リアルタイム進行風)**: 「月曜9時、ファイルサーバーにアクセス不可」から開始。情報収集→影響範囲特定→広報(社内アナウンス文を実際に選ぶ/書く)→復旧判断→再発防止レポートまで。選択によって被害範囲・復旧時間が変動し、最後に行動評価レポートが出る
  - HTML「障害対応の型」: 発生時タイムライン、役割分担(指揮・作業・広報)、障害報告書テンプレート
- **内容トピック**: 影響範囲の即時把握 / 「直す人」と「伝える人」の分離 / 社内アナウンスの書き方 / ポストモーテム文化(個人を責めない)
