Part 2 ネットワーク基礎と実践

2-8 ネットワークトラブルシュート実践

🎯 この章で学ぶこと
  • 「つながらない」を層の下から順に切り分ける型を身につける
  • ping・ipconfig・nslookup・tracert を組み合わせて原因を特定できる

1. 切り分けの型 〜下から順に、4連ping〜

# 「インターネットにつながらない」と言われたら、この順で撃つ
① ping 127.0.0.1        # 自分自身(TCP/IPスタックは生きてる?)
② ping <自分のIP>       # NICは生きてる?(ipconfigで確認してから)
③ ping <デフォルトGW>    # 社内LANの入口までは届く?
④ ping 8.8.8.8          # インターネットにIPで届く?
⑤ ping example.jp       # 名前でも届く?(=DNSの確認)
どこで失敗?疑う場所
①②で失敗端末自身(NIC無効、ドライバー、IP未取得=169.254.x.x)
③で失敗LAN内(ケーブル、スイッチ、VLAN、Wi-Fi、GW設定ミス)
④で失敗出口(ルーター、FW、回線、プロバイダ)
④は成功、⑤で失敗DNS(設定・サーバー障害)← 頻出!
⚠️ 169.254.x.x を見たら: DHCPからIPをもらえなかったサイン(APIPA)。DHCPサーバー・VLAN・ケーブルを疑います。ここ、テストに出ますよ!

2. ラボ: わざと壊して直す(90分)

自宅PCで「壊す→症状を見る→直す」を体験します(すべて元に戻せる操作です)。

  1. DNS故障ごっこ: アダプター設定でDNSサーバーを 127.0.0.2 に手動設定 → ブラウザは全滅、でも ping 8.8.8.8 は通ることを確認(=④○⑤×の状態)→ 自動取得に戻す
  2. hosts汚染ごっこ: hostsに 127.0.0.1 example.jp を追記 → そのサイトだけ開けないことを確認 → 行を削除(「特定サイトだけダメ」の一因を体験)
  3. IP手動ミスごっこ: IPを別サブネットの値に固定 → GWにpingが通らないことを確認 → 自動取得に戻す
  4. 各症状で「4連ping」の結果がどう変わったかを表にまとめる

3. ヒアリングとエスカレーション

  • 聞くべき3点: 「いつから」「あなただけ?周りも?」「直前に何か変えた?」
  • 影響範囲で初動が変わる: 1人だけ→端末から / フロア全体→スイッチ・AP / 全社→出口・回線 (範囲=疑う機器の高さ)
  • 回線業者・保守ベンダーへ渡すとき: 発生時刻、影響範囲、4連pingの結果、直前の変更、を1枚にまとめて渡すと解決が数倍速い
「pingの結果を貼ってもらうだけで、電話の向こうのベンダーさんの声が明るくなる」— 事実の整理は最強の共通言語です。
🎭 ケーススタディで腕試し
3つの「つながらない」事件に、時系列で挑みます。4連pingの型を胸に!
🎭 ケーススタディ「つながらない三重奏」を始める
📌 まとめ
  • 切り分けは下の層から。「4連ping+DNS確認」を体に叩き込む
  • 169.254.x.x=DHCP失敗のサイン
  • 影響範囲が、疑うべき機器の「高さ」を教えてくれる
