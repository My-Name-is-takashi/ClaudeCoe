2-4 DNSの仕組み

🎯 この章で学ぶこと
  • 名前解決の流れ(スタブ→キャッシュ→ルート→権威)を説明できる
  • nslookup / dig で名前解決を調査できる
  • 主要レコード種別(A/AAAA/CNAME/MX/TXT)を読める
📺 動画「DNSはインターネットの電話帳」(13分)を先にどうぞ。

1. DNSは「インターネットの電話帳」

人間は example.co.jp という名前でアクセスしたい。でも通信の宛先はIPアドレス。この変換(名前解決)を担うのがDNSです。Webもメールも、ほぼすべての通信は最初にDNSに聞く——だからDNSが壊れると「インターネット全部が壊れた」ように見えます。

PC(スタブリゾルバー)→キャッシュDNS
(社内 or プロバイダ)
↓ 知らなければ代わりに探し回る(再帰問い合わせ)
ルートDNS
「.jpなら あっちへ」→.jp/.co.jpのDNS→example.co.jpの権威DNS
「答えは 203.0.113.10」
キャッシュDNSが電話帳係。一度調べた答えはTTLの間キャッシュされる。

2. レコード種別 〜電話帳の項目〜

種別意味実務での登場場面
A名前 → IPv4アドレスWebサーバーの引っ越し=Aレコード変更
AAAA名前 → IPv6アドレスIPv6対応サイト
CNAME名前 → 別の名前(別名)SaaSに独自ドメインを向ける設定で頻出
MXこのドメイン宛メールの届け先メール移行で最重要。壊すとメールが消える
TXT自由記述SPF/DKIM(送信ドメイン認証)、ドメイン所有確認
NSこのドメインの権威サーバードメイン移管時に確認
⚠️ TTLとキャッシュ: レコードには TTL(キャッシュしてよい秒数)があります。「DNSを変えたのに切り替わらない」のはキャッシュが残っているから。切り替え作業の前にTTLを短くしておくのがプロの段取りです。

3. ラボ: digで名前解決を追いかける(60分)

# ① まずは基本の問い合わせ(Windowsは nslookup、Linux/Macは dig)
nslookup example.jp
dig example.jp            # ANSWER SECTION の A レコードとTTLを確認

# ② レコード種別を指定して調べる
dig example.jp MX         # メールの届け先
dig example.jp TXT        # SPFなど
dig www.example.jp CNAME

# ③ 名前解決の旅路を全部見る(ルートから順に)
dig +trace example.jp

# ④ 問い合わせ先DNSを指定する(切り分けの必殺技)
dig @8.8.8.8 example.jp   # 社内DNSがおかしい?外のDNSに直接聞いて比較
  1. ①〜④を実行し、それぞれの出力の意味をメモする
  2. hostsファイル(Windows: C:\Windows\System32\drivers\etc\hosts)に
    127.0.0.1 test.example.local を追記(管理者権限)→ ping test.example.local で効くことを確認 → 行を削除して後片付け
  3. ✅ 確認: 「hostsはDNSより先に参照される」ことを自分の言葉で説明できればクリア

4. 「DNSを疑うべき症状」パターン集

  • IP直打ちなら開けるのに、名前だと開けない → ほぼDNS
  • 「特定のサイトだけ」「切り替え直後だけ」おかしい → キャッシュ/TTL
  • 社内サイトだけ引けない → 社内DNSサーバーを疑う(dig @社内DNS と dig @8.8.8.8 を比較)
  • メールだけ届かない → MX/SPFレコードを確認
障害の世界には「とりあえずDNSを疑え」という格言があるくらい、DNS起因は多いんです。`dig @` で聞き先を変える技、必ず手に馴染ませてくださいね。
📌 まとめ
  • DNSは名前→IPの電話帳。壊れると「全部壊れた」ように見える
  • A/CNAME/MX/TXTは実務頻出。TTLを制する者が切り替え作業を制す
  • dig(nslookup)+問い合わせ先指定が切り分けの武器
