Part 7 IT運用とヘルプデスク

7-4 監視とアラート

🎯 この章で学ぶこと
  • 監視の4種類(死活・リソース・ログ・外形)を設計できる
  • 監視ツールを実際に立てて発報を体験する
  • 「アラート疲れ」を起こさない閾値設計を学ぶ

1. 監視の4種類

種類見るもの例気づける障害
死活監視生きてる?ping / ポート応答サーバー・機器のダウン
リソース監視余力ある?CPU / メモリ / ディスク使用率ディスクフル前の予兆(3-12章の悲劇を防ぐ!)
ログ監視変な記録は?エラー多発、認証失敗の連続アプリ異常・攻撃の兆候
外形監視ユーザーから見えてる?URLへのHTTPS応答・応答時間「中は正常なのに外から見えない」(証明書切れ・DNS)

「監視されていないシステムは、止まっていても気づけない」。ユーザーからの「つながらないんですけど」が検知手段になっている状態を卒業するのがこの章です。

2. 閾値設計 〜オオカミ少年を作らない〜

  • 2段階で切る: Warning(翌営業日対応でよい)/ Critical(今すぐ動く)。例: ディスク 80%でW、90%でC
  • 「瞬間値」でなく「継続時間」で発報(CPU 90%が5分継続、など)——スパイクでの誤報を防ぐ
  • アラート疲れが最大の敵: 対応不要な通知が混ざると、本物も無視されるようになる。「鳴ったら必ず何かする」通知だけ残し、それ以外はレポートに落とす
  • 通知先の設計: Critical=電話/プッシュ、Warning=チャット、情報=日次サマリー
💪 ラボ: はじめての監視構築(120分)
  1. mirai-lab01(Part 3のVM)に軽量監視ツール Uptime Kuma を導入(Docker: docker run -d -p 3001:3001 louislam/uptime-kuma ※Docker未導入なら sudo apt install docker.io)
  2. 監視を3つ登録: ①ping(VM自身)②HTTP(http://localhost のnginx)③TCP(22番)
  3. sudo systemctl stop nginx → ダウン検知と通知(画面)を確認 → start で復旧検知
  4. 通知チャネル(メール等)を設定し、実際に通知を受け取る
  5. 発展: 本格ツール(Zabbix)ではリソース監視・閾値・グラフまで扱えることをドキュメントで確認
  6. 設計演習: ミライ商事の資産(AD×2、ファイルサーバー、社内ポータル、VPN装置、各拠点ルーター)に対する監視項目表を作る
⚠️ 監視の監視: 監視サーバー自体が落ちたら全部見えなくなります。外形監視だけは外部サービス(クラウド側)から行う、監視サーバーの死活を別系統で見る、の二重化を忘れずに。
初めて「ダウン通知」が自分のスマホに届いた瞬間、あなたは「気づかされる側」から「気づく側」になります。これ、情シスとしてすごく大きな一歩なんですよ。
📌 まとめ
  • 監視は死活・リソース・ログ・外形の4点セット。特にディスクは予兆で捕まえる
  • 閾値は2段階+継続時間。「鳴ったら必ず動く」通知だけを残す
  • 監視サーバー自身の監視(外部からの外形)を忘れない
