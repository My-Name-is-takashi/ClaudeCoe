3-12 ログの読み方とトラブルシュート

🎯 この章で学ぶこと
  • /var/log と journalctl から障害原因を特定する型を身につける
  • nginxアクセスログのフォーマットを読解できる
  • logrotate によるログ肥大対策を知る

1. ログの地図 〜どこに何が書かれるか〜

場所内容見るとき
journalctl -u サービス名各サービスの動作ログサービスが落ちた・上がらない
/var/log/syslogシステム全般「何が起きたか分からない」ときの起点
/var/log/auth.logログイン・sudoの記録不正アクセス調査・操作の追跡
/var/log/nginx/access.logWebへの全アクセス誰が何を見たか、攻撃の痕跡
/var/log/nginx/error.lognginxのエラー502・SSL・設定ミスの調査

2. アクセスログの読解(1行を分解する)

192.168.10.5 - - [06/Jul/2026:09:12:01 +0900] "GET /index.html HTTP/1.1" 200 5120 "-" "Mozilla/5.0 ..."
└─誰が      └─いつ                        └─何を          └─結果 └─サイズ      └─何のブラウザで
コード意味代表的な原因
200成功正常
301/302リダイレクトhttp→https誘導など(正常なことが多い)
403アクセス拒否パーミッション(3-6章!)・拒否設定
404存在しないパス間違い・消したファイルへのリンク
500サーバー内エラーアプリのバグ・設定ミス
502不正なゲートウェイ背後のアプリ(PHP-FPM等)が死んでいる
503過負荷・メンテ中リソース枯渇・意図的な停止

3. 調査の型 〜5ステップ〜

① 症状の時刻を特定する(「いつから?」)
② その時刻のログを絞り込む
   journalctl --since "09:00" --until "09:30"
   grep "06/Jul/2026:09:1" access.log
③ エラーだけ抽出する
   grep -iE "error|crit|fail" error.log | tail -20
④ 資源を確認する(ログに答えがない時)
   df -h(ディスク)/ free -h(メモリ)/ top(CPU)
⑤ エラーメッセージ原文で検索する(最強の武器)

4. logrotate 〜ログに食い尽くされないために〜

ログは放置すると無限に太り、ディスクフル→サービス全停止という最悪の障害を起こします。Ubuntuでは logrotate が日次で圧縮・世代管理していますが、自作アプリのログは対象外。/etc/logrotate.d/ に設定を足すのが運用者の責任です。

# /etc/logrotate.d/myapp の例
/var/log/myapp/*.log {
    daily           # 毎日ローテート
    rotate 14       # 14世代保持
    compress        # 圧縮する
    missingok       # 無くてもエラーにしない
    notifempty      # 空ならスキップ
}

📌 まとめ
