Part 3 Linuxサーバー入門〜実践

3-6 ユーザー・グループとパーミッション

🎯 この章で学ぶこと
  • rwx 表記と数値表記(755/644/600)を相互変換できる
  • グループを使ったファイル共有権限を設計・実装できる
  • sudo の仕組みと作法を理解する

1. ls -l の読み方 〜9文字の暗号を解く〜

-rw-r--r-- 1 mirai keiri 1024 Jul  6 10:00 houkoku.txt
│└┬┘└┬┘└┬┘   └─所有者  └─グループ
│ │  │  └ その他(o): r--(読みのみ)
│ │  └ グループ(g): r--(読みのみ)
│ └ 所有者(u): rw-(読み書き)
└ 先頭: - 普通のファイル / d ディレクトリ / l リンク
記号数値ファイルでの意味ディレクトリでの意味
r4読める一覧(ls)できる
w2書き換えられる中にファイルを作成・削除できる
x1実行できる中に入れる(cd できる)
755 = rwx r-x r-x   # 定番: 公開ディレクトリ・スクリプト
644 = rw- r-- r--   # 定番: 一般ファイル・設定ファイル
600 = rw- --- ---   # 定番: 秘密のファイル(SSH秘密鍵はコレ)
⚠️ 777は禁止ワード: 「動かないから777にした」は誰でも書き換え可能=改ざんし放題の状態。原因を調べずに全開放するのは、玄関の鍵を外して「開かない問題」を解決するのと同じです。

2. 変更コマンド3兄弟

chmod 644 houkoku.txt            # 権限変更(数値指定)
chmod g+w houkoku.txt            # 記号指定: グループに書き込みを追加
chown mirai:keiri houkoku.txt    # 所有者:グループを変更(root権限が必要)
chgrp keiri houkoku.txt          # グループだけ変更

3. グループで共有を設計する

# 「経理チームは読み書き、他部署は閲覧のみ」の共有ディレクトリ
sudo groupadd keiri                      # グループ作成
sudo useradd -m -G keiri tanaka          # ユーザー作成+keiriに所属(-m ホーム作成)
sudo mkdir /srv/keiri-share
sudo chown root:keiri /srv/keiri-share
sudo chmod 775 /srv/keiri-share          # 所有者rwx / keiri rwx / 他は r-x
# 確認: id tanaka(所属グループ)、ls -ld /srv/keiri-share

4. sudoの作法

  • sudo は「この1コマンドだけ管理者として実行」。rootで常時ログインしないのが鉄則(誤操作が即、全損になるため)
  • 誰がsudoを使えるかは /etc/sudoers(Ubuntuでは sudo グループ所属)で管理
  • sudoの実行は記録される(/var/log/auth.log)——「誰が何をしたか」が追える状態を保つのも情シスの仕事
💪 ラボ: 権限設計演習(75分)
  1. 上記の手順で keiri グループと共有ディレクトリを作る
  2. ユーザー tanaka(keiri所属)と suzuki(未所属)を作成
  3. su - tanaka で成り代わり、共有ディレクトリにファイルを作成できることを確認
  4. su - suzuki では閲覧はできるが作成できないことを確認(エラーメッセージを記録)
  5. 「suzukiも書けるようにして」と言われたら、chmodではなくグループ追加(sudo usermod -aG keiri suzuki)で対応するのが正道であることを確認
  6. 後片付け: 作成したユーザーとディレクトリを削除(userdel -r)、またはスナップショットへ復帰
「777にしたら動きました!」は情シス見習い最大の罠。動いた理由を説明できない設定は、動いていても事故です。グループ設計で解くクセをつけましょうね。
📌 まとめ
  • 9文字は u/g/o × rwx。755・644・600は暗記でOK
  • 共有は「グループを作って所属させる」が正道。777は禁じ手
  • rootに成らずsudoで。操作の記録が残る形を保つ
