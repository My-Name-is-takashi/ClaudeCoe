3-10 nginx入門:インストールと設定の読み方

  • server / location のマッチングを理解し、バーチャルホストを設定できる
  • リバースプロキシとHTTPS化の設定を読み書きできる
  • 安全な変更手順「nginx -t → reload」を習慣化する
📺 動画①「nginxってなに?」②「nginx.conf 読解ツアー」とあわせてどうぞ。

1. nginxの2つの顔

役割何をする?ミライ商事での例
WebサーバーHTMLや画像などの静的ファイルを配る社内ポータルのページ配信
リバースプロキシ「受付窓口」として手前に立ち、背後のアプリへ転送する勤怠アプリ(内部の8080番)を443番で受けて転送

リバースプロキシを挟む理由: ①HTTPS化を一手に引き受ける ②複数アプリを1つの入口に集約 ③アプリを直接インターネットに晒さない。現代のサーバー構成の定番形です。

2. 設定ファイルの地図

/etc/nginx/
├── nginx.conf              # 大元。全体設定と include の起点
├── conf.d/*.conf           # 追加設定の置き場(includeされる)
├── sites-available/        # サイト定義の「倉庫」(Ubuntu流)
│   └── default
└── sites-enabled/          # 「有効化」されたサイト(availableへのシンボリックリンク)
    └── default -> ../sites-available/default

Ubuntu流の運用: 設定は sites-available に書き、ln -s でsites-enabled に置いた分だけ有効。無効化はリンクを消すだけなので、切り戻しが安全にできます。

3. nginx.conf の文法は「2つだけ」

# ① ディレクティブ: 「名前 値;」— セミコロン必須!
worker_processes auto;

# ② ブロック: 「名前 { … }」— 中に設定を入れ子にできる
http {
    server {                          # server = 1つのサイト(バーチャルホスト)
        listen 80;                    # 待ち受けポート
        server_name portal.mirai.example.co.jp;   # どのホスト名宛を受けるか
        root /var/www/portal;         # ドキュメントルート
        index index.html;

        location / {                  # location = URLパスごとの設定
            try_files $uri $uri/ =404;
        }
        location /api/ {              # /api/ 宛だけ別の動きをさせる
            proxy_pass http://127.0.0.1:8080;
        }
    }
}

読み方のコツ: 外側から「http → server(どのサイト?)→ location(どのURL?)」と絞り込む。逆に言えば、どんな複雑な設定もこの3段の入れ子です。

4. locationマッチの優先順位(頻出!)

書き方意味優先度
location = /login完全一致1位(即決)
location ^~ /static/前方一致(正規表現より優先の宣言)2位
location ~ \.php$正規表現(~* は大文字小文字無視)3位(定義順)
location /前方一致(最長一致で選ばれる)4位(受け皿)

例: /static/app.php へのアクセスは——^~ /static/ があればそちらに決まり、正規表現 \.php$ は評価されません。「このURLはどのlocationに当たる?」を読めることが、nginx読解力の核心です。

5. リバースプロキシの定番セット

location / {
    proxy_pass http://127.0.0.1:8080;      # 背後のアプリへ転送
    proxy_set_header Host $host;            # 元のホスト名をアプリに伝える
    proxy_set_header X-Real-IP $remote_addr;         # 本当の客のIPを伝える
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;      # http/httpsの別を伝える
}
⚠️ 末尾スラッシュの罠: proxy_pass http://127.0.0.1:8080/;(スラッシュあり)と …:8080(なし)では転送されるパスが変わります。「あり」はlocation部分を差し替え、「なし」はそのまま渡す。プロキシが404になったらまずここを疑いましょう。

6. HTTPS化と安全な変更手順

server {
    listen 443 ssl;
    server_name portal.mirai.example.co.jp;
    ssl_certificate     /etc/nginx/ssl/portal.crt;   # 証明書
    ssl_certificate_key /etc/nginx/ssl/portal.key;   # 秘密鍵(600!)
    ...
}
server {
    listen 80;
    server_name portal.mirai.example.co.jp;
    return 301 https://$host$request_uri;   # httpは httpsへ恒久リダイレクト
}
# 変更のたびに必ずこの3点セット(息をするように)
sudo nginx -t                    # ① 構文チェック(syntax is ok / test is successful)
sudo systemctl reload nginx      # ② 無停止で反映
curl -I http://localhost         # ③ 動作確認(ステータスコードを見る)
💪 ラボ: nginx設定演習(150分)
  1. スナップショット取得 → nginx導入済みの mirai-lab01 で /etc/nginx/ の構造を ls -R と cat で一巡する
  2. grep -v "^\s*#" /etc/nginx/nginx.conf | grep -v "^$" でコメントを除いた「本体」を読む(3-4章の技!)
  3. 静的サイト追加: /var/www/portal/index.html を作り、sites-available に portal 定義を書き、シンボリックリンクで有効化 → curl -H "Host: portal.mirai.example.co.jp" http://localhost で確認
  4. locationマッチ実験: = /exact、^~ /static/、~ \.txt$、/ の4つを定義し、それぞれ別の文字列を返すようにして、どのURLがどこに当たるかを予想→curlで答え合わせ
  5. リバースプロキシ: python3 -m http.server 8080 を起動し、/app/ 宛を8080へproxy_pass。末尾スラッシュあり/なしの挙動差を体験
  6. 自己署名証明書でHTTPS化(openssl req -x509 -newkey rsa:2048 -nodes ...)+80→443リダイレクト
  7. 全ステップで「nginx -t → reload → curl」の3点セットを実施
nginxの設定が読めるようになると、Webの障害対応で「どこまでは正常か」を一人で切り分けられるようになります。3-12章のログ調査とあわせて、Part 3のクライマックスですよ!
📌 まとめ
  • 文法は「ディレクティブ;」と「ブロック{}」の2つ。http→server→locationの3段絞り込み
  • locationは = → ^~ → 正規表現 → 前方一致の順で決まる
  • proxy_passは末尾スラッシュに注意。変更は「-t → reload → curl」の3点セット
