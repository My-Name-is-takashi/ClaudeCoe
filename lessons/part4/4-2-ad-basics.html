Part 4 Windows ServerとActive Directory

4-2 Active Directory基礎とドメイン構築

🎯 この章で学ぶこと
  • ドメイン・OU・ユーザー・グループの階層を設計図として描ける
  • 検証ドメインを構築し、クライアントをドメイン参加させられる
  • AGDLP(グループ設計の原則)を説明できる
📺 動画「Active Directoryは社員名簿+鍵束」(13分)を先にどうぞ。
ラボ環境: VirtualBox+Windows Server 2022評価版(180日)+Windows 11クライアントVM。メモリ8GB以下のPCは「画面写真でたどる紙上ラボ」(付録)で代替可。

1. ADは「社員名簿+鍵束」

  • 認証(Authentication): 「あなたは本当に未来さん?」— ログオン時にドメインコントローラー(DC)が確認
  • 認可(Authorization): 「未来さんは経理フォルダーを開いていい?」— グループ所属で判断
  • この2つを全社で一元管理するのがAD。1回のログオンで社内の許されたリソース全てに入れる(シングルサインオンの社内版)
ドメイン mirai.local
OU: 東京
├ 営業部 ├ 経理部OU: 大阪
├ 営業部OU: サーバー/PC
OU(組織単位)=名簿のフォルダー分け。GPO(4-3章)を当てる単位になるのが最重要ポイント。
用語意味
ドメイン管理の単位(例: mirai.local)。1社1ドメインが基本形
DC(ドメインコントローラー)ADのデータベースを持つサーバー。必ず2台以上で冗長化が実務の鉄則
OUユーザー・PCを整理するフォルダー。GPOのリンク先
KerberosADの認証プロトコル。「チケット」方式(概念だけでOK)
⚠️ ADとDNSは一心同体: クライアントは「DCの居場所」をDNSで探します。クライアントのDNS参照先は必ず社内DCに向けること。「ドメイン参加できない」「ログオンが遅い」の原因第1位はDNS設定ミスです。ここ、テストに出ますよ!

2. AGDLP 〜グループ設計の原則〜

A(Account)   : ユーザーを
G(Global)    : 部署グループに入れる       例: G-経理部
DL(DomainLocal): リソース用グループに入れる 例: DL-経理共有-書込
P(Permission): DLグループに権限を付ける    例: フォルダーACLにDL-経理共有-書込

「ユーザーに直接権限を付けない」のが要点。人事異動が「Gグループの入れ替えだけ」で完結し、権限の棚卸しも一目でできます(4-4章・4-5章で実践)。

3. ラボ: ミライ商事ドメイン構築(150分)

  1. Windows Server 2022 評価版VMを作成(メモリ4GB/ディスク60GB)→ 固定IP設定・スナップショット
  2. サーバーマネージャー→「役割と機能の追加」→ AD DS を追加
  3. 「このサーバーをドメインコントローラーに昇格」→ 新しいフォレスト mirai.local → DSRMパスワード設定 → 再起動(DNSも自動で入る)
  4. 「Active Directory ユーザーとコンピューター」で OU設計(東京/大阪 × 営業部/経理部/情シス)を作成
  5. ユーザー mirai-taro(営業部)等を3名、グループ G-eigyo G-keiri を作成し所属させる
  6. Windows 11 VMのDNSをDCのIPに向け、「設定→システム→ドメイン参加」で mirai.local に参加 → 再起動
  7. mirai\mirai-taro でログオンできれば成功!DC側の「AD ユーザーとコンピューター」にPCが登録されたことも確認
💡 うまくいかない時(定番3つ)
  • 「ドメインが見つからない」→ クライアントのDNSがDCを向いているか(nslookup mirai.local で確認)
  • 時刻ズレ → Kerberosは時刻差5分以上で認証失敗。両VMの時刻を確認
  • ネットワーク種別 → VM同士が同じホストオンリーネットワークにいるか
自分のドメインに初めてログオンできた瞬間は、Part 3のSSH初成功と並ぶ「情シス2大感動体験」です。おめでとうございます!
📌 まとめ
  • AD=社員名簿+鍵束。認証と認可を全社で一元化
  • OUはGPOを当てる単位。DCは2台以上、DNSとは一心同体
  • 権限はAGDLP。ユーザーに直接権限を付けない
