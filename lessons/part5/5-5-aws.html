Part 2〜4で学んだ概念は、名前を変えてAWSに全部再登場します。Azureも同様(VPC=VNet、EC2=Virtual Machines、IAM≒Entra ID+RBAC)。

2. 事故らないための3大ルール

⚠️ 学習前に必ず設定する3つ
  • ① 予算アラート: Billing→Budgetsで「月500円超えたらメール」を最初に設定(青天井課金の防止)
  • ② ルートユーザーを封印: MFAを付けて日常利用禁止。作業はIAMユーザーで(root常用禁止はLinuxと同じ思想)
  • ③ アクセスキーを漏らさない: GitHub等に上げた瞬間、数分でマイニングに悪用され数十万円の請求が来る——実在する定番事故です

3. S3公開事故 〜情シスが知るべき代表的事故〜

S3バケットの公開設定ミスによる情報漏えいは、世界中で繰り返されてきた定番事故。現在は「ブロックパブリックアクセス」が既定で有効です。「既定でオンの安全装置を、理由なく外さない」——責任共有モデル(5-1章)の実践がここでも問われます。

💪 ラボ: AWS最小構成構築(180分・無料枠)
  1. アカウント作成 → 最初に予算アラート設定 → ルートにMFA → 管理用IAMユーザー作成(AdministratorAccess+MFA)
  2. IAMユーザーでログインし直す(以後ルートは使わない)
  3. VPC作成: 10.0.0.0/16、パブリックサブネット 10.0.1.0/24(Part 2-2の計算がそのまま!)
  4. EC2起動: Ubuntu 24.04 / t2.micro(無料枠)/ キーペア作成 → セキュリティグループはSSH(22)を「自分のIPのみ」許可
  5. 手元からSSH接続(3-9章の技!)→ sudo apt update && sudo apt install nginx → セキュリティグループに80を追加してブラウザで確認(3-7章!)
  6. S3バケット作成 → ファイルをアップロード → 公開ブロックが効いていることを確認
  7. 後片付け(超重要): EC2終了 → EIP解放 → S3バケット削除 → 翌日Billingで課金ゼロを確認
手順5で「SSHで入ってnginxを立てる」——Part 3と同じことをクラウドでやっただけ、と気づきましたか?基礎は裏切らないんです。
📌 まとめ
  • AWSはオンプレ知識の翻訳で理解できる(VPC=LAN、EC2=VM、IAM=AD)
  • 予算アラート・ルート封印・キー厳守の3大ルールを最初に
  • セキュリティグループは最小開放、S3の公開ブロックは外さない、使ったら消す
